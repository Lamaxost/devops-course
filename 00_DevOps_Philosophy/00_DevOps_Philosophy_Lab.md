# Тема 0: Навіщо це все? Філософія та принципи DevOps — Лабораторна робота

> **Файл для студентів.** Практична частина до лекції `00_DevOps_Philosophy_Lecture.md`.

---

## 🎯 Мета роботи

Дослідити реальний DevOps-проєкт зсередини: зрозуміти, як принципи CALMS проявляються в конкретних файлах, папках і командах реального репозиторію. Після виконання цієї роботи ви зможете читати структуру DevOps-репозиторію і розуміти, *чому* він організований саме так.

> **Що потрібно:** комп'ютер з Git та термінальним емулятором. Інтернет-доступ.

---

## 🛠 Покрокова інструкція

### Крок 1: Клонуємо репозиторій проєкту

Ми будемо досліджувати реальний проєкт AI-асистента, побудований викладачем. Він є нашим навчальним полігоном на весь курс.

```bash
# Клонуємо репозиторій у папку my-ai-assistant
git clone https://github.com/sobol-mo/My_AI_Assistant.git my-ai-assistant

# Переходимо в папку проєкту
cd my-ai-assistant
```

**Очікуваний результат:** У поточній теці з'явиться папка `my-ai-assistant` із файлами проєкту.

---

### Крок 2: Досліджуємо структуру репозиторію

```bash
# Переглядаємо корінь репозиторію
ls -la

# Переглядаємо дерево папок верхнього рівня
find . -maxdepth 2 -type d | sort
```

**Очікуваний результат:** Ви побачите, що репозиторій організований за папками з числовими префіксами (`00_`, `01_`, `10_`). В одному репозиторії — і бізнес-вимоги, і архітектура, і код, і інфраструктурні скрипти. Це і є "Single Source of Truth".

---

### Крок 3: Знайдіть "Pets vs. Cattle" в реальних файлах

Відкрийте файл, де описана конфігурація сервера:

```bash
# Переглядаємо Ansible playbook для розгортання агента
cat 10_Implementation/openclaw/ansible/playbook-openclaw-setup.yml
```

**Очікуваний результат:** Ви побачите YAML-файл із інструкціями, які автоматично налаштовують сервер. Це і є Infrastructure as Code: сервер описаний як **"Cattle"** (відтворюваний із файлу), а не **"Pet"** (налаштований вручну і незамінний).

---

### Крок 4: Знайдіть принцип Least Privilege у конфігурації

```bash
# Знайдемо, де в playbook описується створення користувача
grep -A 5 "openclaw" 10_Implementation/openclaw/ansible/playbook-openclaw-setup.yml | head -40
```

**Очікуваний результат:** Ви побачите, що агент запускається від імені окремого системного користувача `openclaw` без прав `sudo`. Це реалізація принципу **Principle of Least Privilege** — навіть якщо агента зламають, зловмисник не отримає доступ до всього сервера.

---

### Крок 5: Дослідіть гілки репозиторію — стратегія

```bash
# Переглядаємо всі гілки репозиторію
git branch -a

# Читаємо документ з описом стратегії гілкування
cat AGENT.md | grep -A 30 "Branching"
```

**Очікуваний результат:** Ви побачите кілька гілок: `main`, `python`, `ClawdBot`. Кожна гілка — це окремий "технологічний стек" або середовище. Гілка `main` містить лише технологічно-нейтральну архітектуру (документи та схеми). Це і є DevOps-підхід до version control: **код і інфраструктура еволюціонують разом, прозоро та відстежувано**.

---

### Крок 6: Визначте CALMS у проєкті

Відкрийте файл `DEVLOG.md` і перегляньте 2–3 останніх записи:

```bash
# Переглядаємо лог розробки (останні 80 рядків)
tail -80 DEVLOG.md
```

**Очікуваний результат:** Ви побачите хронологічні записи про рішення, проблеми та зміни в проєкті. Це реалізація **Sharing** (принцип S у CALMS) — всі рішення зафіксовані та доступні для команди.

---

## ✅ Результат виконання роботи

Після виконання всіх кроків ви повинні вміти відповісти: де в цьому репозиторії живуть принципи **Automation**, **Sharing** та **Measurement**? — і показати конкретні файли.

```bash
# Фінальна перевірка: переконайтеся, що ви знаходитеся в правильній папці
pwd
git log --oneline -5
```

**Очікуваний результат:** Виводяться останні 5 комітів проєкту. Зверніть увагу на формат повідомлень до комітів (`docs:`, `feat:`, `fix:`) — це **Conventional Commits**, стандарт, який робить DORA-метрику Lead Time вимірюваною.
