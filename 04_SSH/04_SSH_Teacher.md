## 💬 Питання для обговорення: еталонні відповіді

### Питання 1

> _Ваш колега каже: «Навіщо морочитися з ключами? Я просто вимкну вимогу до пароля і буде зручніше». Що принципово не так з цим підходом?_

**Еталонна відповідь:**

Вимкнути автентифікацію взагалі — означає залишити двері відчиненими для всіх. SSH-порт (22) сканується ботами по всьому Інтернету цілодобово. Без автентифікації будь-хто, хто знайде IP сервера, отримує доступ до системи без жодних перешкод.

Ключова різниця між паролем і ключем — не в зручності, а в безпеці:

- **Пароль** можна підібрати (brute force), вкрасти через фішинг або перехопити при введенні. Автоматизований скрипт не може безпечно зберігати й вводити пароль.
- **SSH-ключ** математично неможливо підібрати за розумний час. Ключ ніколи не «вводиться» і не передається через мережу. Навіть якщо трафік перехоплений — з нього неможливо відновити ключ.

Правильна відповідь колезі: «Ти вимкнеш пароль — але залишиш ключ. Саме ключ замінює пароль, а не скасовує автентифікацію».

**Методичний коментар:** Гарна нагода для дискусії. Запитайте студентів: «А що буде, якщо вимкнути і пароль, і ключ?» → відповідь: SSH взагалі не підніметься або буде вразливий до підключення будь-кого.

### Питання 2

> _Після `ssh-copy-id` ви підключаєтесь до VM без пароля. Потім переустановлюєте VM з нуля. Ви пробуєте підключитись — SSH видає `WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!`. Чому так сталося і що треба зробити?_

**Еталонна відповідь:**

При першому підключенні SSH зберіг «відбиток» сервера (host key) у файл `~/.ssh/known_hosts` на вашому комп'ютері. Цей відбиток — унікальний цифровий підпис, який генерується при встановленні SSH-демона на сервері.

Коли VM перевстановили — SSH-демон згенерував **нові** ключі. Відбиток сервера змінився, але у `known_hosts` залишився старий. Тому SSH піднімає тривогу: «Я знаю цю адресу, але сервер відповідає іншим підписом — можливо, це атака MITM».

**Що треба зробити:**

1. Переконатись, що причина зміни відома (саме перевстановлення, а не атака).
2. Видалити старий запис з `known_hosts`:
   ```bash
   ssh-keygen -R 192.168.56.10
   ```
3. Підключитись знову та підтвердити новий відбиток.

**Методичний коментар:** Підкресліть, що ігнорувати це попередження на Production — небезпечно. Тут — ми самі перевстановили VM, тому знаємо причину. В реальній роботі треба завжди з'ясовувати, чому відбиток змінився.

### Питання 3

> _Ви хочете дати доступ до VM ще двом колегам. Чи потрібно їм надсилати ваш приватний ключ? Як правильно організувати спільний доступ?_

**Еталонна відповідь:**

Ні, приватний ключ **ніколи** не передається іншим. Це порушує саму концепцію асиметричної криптографії. Якщо ключ скомпрометований — змінювати його доведеться на всіх серверах, де він використовувався.

**Правильний підхід:**

Кожен колега генерує **власну** пару ключів на своєму комп'ютері:
```bash
ssh-keygen -t ed25519 -C "colleague@example.com"
```

Потім надсилає вам свій **публічний** ключ (`id_ed25519.pub`). Ви додаєте його до `~/.ssh/authorized_keys` на сервері — новим рядком:

```bash
echo "ssh-ed25519 AAAA... colleague@example.com" >> ~/.ssh/authorized_keys
```

Тепер кожен має власний ключ, а ви маєте повний контроль: хочете забрати доступ у конкретного колеги — видаляєте його рядок з `authorized_keys`. Ваш ключ при цьому не чіпається.

**Методичний коментар:** Це відповідає принципу найменших привілеїв та аудитованості. Якщо всі використовують один ключ — неможливо визначити, хто саме і коли підключався. «Один ключ — один доступ» уможливлює індивідуальний аудит.

### Питання 4

> _SSH-тунель запущений: `ssh -L 8080:localhost:18789 openclaw@192.168.56.10`. Ви відкриваєте `http://localhost:8080` у браузері — і бачите UI OpenClaw. Потім закриваєте термінал з SSH. Що станеться з браузерною вкладкою? Чому?_

**Еталонна відповідь:**

Браузерна вкладка «зависне» або видасть помилку підключення при наступному запиті (наприклад, при оновленні сторінки).

**Чому:** SSH Local Port Forwarding — це не самостійний сервіс. Це «труба», яка існує рівно стільки, скільки існує SSH-сесія. Коли термінал закрито — SSH-процес завершився, «труба» зникла. Порт `8080` на вашому ноутбуці більше нікуди не веде.

Поточна вкладка може ще відображати вміст (якщо він завантажений у кеш браузера), але будь-який новий запит до `localhost:8080` буде відхилений з помилкою `Connection refused`.

**Що робити, якщо потрібен постійний тунель:**

- Задача `ssh -NL 8080:localhost:18789 openclaw@192.168.56.10 &` (фон, без інтерактивної сесії)
- Або записати `LocalForward` у `~/.ssh/config` — тоді тунель відкривається автоматично при кожному підключенні

**Методичний коментар:** Це питання виявляє розуміння того, що тунель — це не маршрут, а активне з'єднання. Аналогія: якщо прибрати шланг між двома відрами — вода не потече, навіть якщо відра ще стоять на місці.

## 📝 Методичні нотатки

### Ключові «моменти розуміння» цієї теми

1. **«Ключ і замки»** — студенти добре сприймають метафору одного ключа і кількох замків. Найчастіша помилка розуміння: думають, що «публічний ключ» = щось публічне в сенсі «відоме всім» і небезпечне. Поясніть: публічний = можна роздавати, але без приватного ключа — марний.

2. **Bootstrap через `vagrant`** — студенти дивуються, чому не використовуємо `ssh-copy-id`. Ключовий інсайт: `openclaw` не має пароля (`!` в `/etc/shadow`), тому перший крок — завжди через вже існуючий привілейований канал. Це реальна DevOps-практика.

3. **Тунель = активне з'єднання** — не маршрут і не налаштування мережі. Це найчастіша концептуальна помилка. Питання 4 прямо перевіряє це розуміння.

### Типові помилки студентів у лабораторній

| Симптом | Причина | Вирішення |
| ------- | ------- | --------- |
| `Permission denied (publickey)` | Неправильні права на `~/.ssh/` або `authorized_keys` | `chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys` |
| `WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!` | VM перестворена, старий ключ у `known_hosts` | `ssh-keygen -R 192.168.56.10` |
| `This account is currently not available` | Shell `/usr/sbin/nologin` | `sudo chsh -s /bin/bash openclaw` |
| Тунель є, але сайт не відкривається | Сервер не запущений на `127.0.0.1:18789` | Запустити `python3 -m http.server 18789 --bind 127.0.0.1` |
| `bind: Address already in use` | Локальний порт 8080 вже зайнятий | Змінити на інший порт: `-L 8081:localhost:18789` |
